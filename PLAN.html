RTG Frontend Scope

The limitation sweep is resolved only in the diagnostic sense. Ordinary Go forms currently rejected by the frontend are in scope and must be implemented over time unless they are listed in the explicit exclusions below. The exclusion list is closed: every non-excluded Go feature belongs in the frontend plan, even when the current implementation only has a diagnostic or a narrow lowering for that form.

Existing diagnostics or partial implementations for maps, interfaces, arrays, function values, dynamic dispatch, defer, panic, recover, and other ordinary Go features are temporary implementation gaps. They must not be treated as language policy exclusions.

For avoidance of doubt, defer, panic, and recover are required frontend semantics. The frontend scope is ordinary Go minus only goroutines, channels, select, cgo, and generics.

Backend omissions in SUBSET.md do not exclude a feature from the frontend. When a normal Go construct is not part of the backend subset, the frontend should parse, check, and lower it into platform-independent units unless that construct is one of the explicit exclusions below.

Explicit Exclusions

This is the closed list of unsupported Go features for now. There are no other language exclusions. Everything else, including runtime control-flow semantics such as defer, panic, and recover, should be treated as frontend work, with backend changes required only when the feature is added to SUBSET.md.

  • goroutines
  • channels
  • select
  • cgo
  • generics

Backend Boundary

If a feature is added to SUBSET.md, it becomes a backend compiler requirement. Otherwise, implement it in rtg/ by lowering ordinary Go into stable platform-independent units before it reaches the backend.
