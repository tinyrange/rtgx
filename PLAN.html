RTG Frontend Rebuild Plan

The old rtg/ frontend was removed because it grew into a collection of feature-specific rewrites that could pass local examples without remaining a useful compiler. The replacement should be built as a small compiler pipeline with explicit contracts, stage gates, and stable units from the start.

Lessons To Integrate

  • Tests must lead the implementation. The rtg_tests/ corpus is the acceptance contract; quick tests should always run, extended tests can remain opt-in.
  • Self-hosting cannot be treated as a late milestone. The frontend should pass hello-world, package loading, quick corpus, stage1, stage2, and stage3 gates in that order.
  • Partial erasure is not feature support. A feature is supported only when the compiler has typed semantics, stable lowering, diagnostics for unsupported edge cases, and tests.
  • The backend subset stays small. Ordinary Go features outside SUBSET.md belong in the frontend unless they are one of the closed exclusions: goroutines, channels, select, cgo, or generics.
  • The unit format must be real infrastructure, not a side effect of file emission. Units should be deterministic, binary, distributable, and usable in memory by a separate linker stage.
  • The package and module model must come before broad language work. Import resolution, package initialization, symbol ownership, and linker boundaries are the shape every later feature must fit.
  • Runtime intrinsics should stay hidden behind lowering. User-facing target-specific escape hatches such as syscalls and static links should not leak into portable frontend semantics.
  • Do not grow rtg/std speculatively. Add compatibility packages only when compiler code, real example code, or an acceptance test benefits from them.
  • Every miscompile gets a minimal reproducer before feature work continues. Regressions belong in tests/ for backend bugs or rtg_tests/ for frontend bugs.
  • Performance gates should exist early. The frontend should track wall time and max RSS for self-hosting before complexity hides the source of regressions.

Architecture

  1. Loader: resolve go.mod, package directories, imports, files, build tags needed by RTG, and deterministic package order.
  2. Parser: accept ordinary Go syntax for the in-scope language and produce precise diagnostics for excluded forms.
  3. Checker: assign types, constants, methods, package symbols, and initialization order without depending on the host Go toolchain at runtime.
  4. Lowerer: translate typed Go into a small platform-independent unit model that can express all frontend-supported features over the backend subset.
  5. Unit encoder: write and read deterministic binary units with stable symbol names, relocations, metadata, and versioning.
  6. Linker: combine package units in memory, resolve symbols, order initializers, include runtime support, and emit one backend input unit.
  7. Driver: provide rtg ./cmd/app, -t os/arch, output selection, diagnostics, and cache controls without changing frontend semantics.

First Bridge

The first implementation step is a parsed source-to-unit bridge for the existing backend subset. A host-Go converter reads current backend test programs, writes a dense TLV unit, and the backend accepts that unit through its compiler input path. This is not the final frontend, but it makes the unit format executable immediately and gives the next compiler a concrete binary contract to emit.

The initial unit carries a token text pool, backend token table, top-level declaration table, and function table. The current backend decodes those binary tables directly into its program representation, skipping source scanning and top-level parsing while keeping the existing expression, statement, type, and code-generation machinery. Later milestones can replace token ranges with denser typed statement and expression nodes without changing the outer unit contract.

Milestones

  1. Define the binary unit format and linker API before implementing broad language features.
  2. Build the minimal loader, parser, checker, lowerer, linker, and driver needed for hello world.
  3. Compile one package with multiple files and deterministic package initialization.
  4. Compile a small imported package graph into separate units, then link it in memory.
  5. Pass the quick rtg_tests/ corpus through the frontend and backend.
  6. Reach stage1 self-hosting with the smallest frontend that can compile itself.
  7. Reach stage2 and stage3 self-hosting with byte-for-byte stable or intentionally explained output differences.
  8. Enable performance gates for frontend self-hosting and keep them visible in CI.
  9. Expand toward the extended corpus by feature group, with each group adding diagnostics, implementation, and tests together.

Feature Policy

The frontend target is ordinary Go minus goroutines, channels, select, cgo, and generics. Maps, interfaces, arrays, function values, dynamic dispatch, defer, panic, recover, complex numbers, unsafe intrinsics, package initialization, import forms, and common builtins are in scope.

Features should be added vertically: syntax, type checking, lowering, runtime/linker support, diagnostics, and tests should land together for a narrow slice. Avoid broad placeholder implementations that only erase unsupported values or special-case one source shape.

Validation

  • go test ./rtg_tests remains the fast frontend corpus check.
  • Extended corpus tests remain environment-gated until they are cheap enough to run routinely.
  • Top-level backend harness tests remain the authority for backend behavior and cross-target output.
  • Frontend self-hosting should report stage, wall time, max RSS, output size, and target set.
  • CI should run the quick frontend corpus, backend tests on supported runners, and the current self-hosting gate once the replacement frontend exists.
