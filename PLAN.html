RTG Frontend Pass Plan

This pass builds the replacement frontend in rtg/. The end goal is concrete: every test in rtg_tests must compile and run through the frontend, and rtg/cmd/rtg must self-host through stage3 using the RTGX backend.

Definition Of Done

  • go test ./rtg_tests passes with the host-built frontend.
  • RTG_FRONTEND_EXTENDED_TESTS=1 go test ./rtg_tests passes with the host-built frontend.
  • Stage0 host build compiles stage1, stage1 compiles stage2, and stage2 compiles stage3.
  • The stage3 frontend can compile rtg_tests/quick and rtg_tests/extended.
  • TestFrontendCompilerPerformance passes: best of three stage2-to-stage3 builds within 500ms wall time, 32MB max RSS, and 1MB output size.
  • The backend tests and backend performance gates still pass.

Hard Constraints

  • The frontend source must be written so it can be compiled by RTGX. Do not depend on the host Go standard library in frontend runtime code.
  • Use local packages under rtg/ and add rtg/std compatibility only when frontend source or an acceptance test directly benefits from it.
  • The frontend emits binary RTG units. It must not emit Go source as its backend contract.
  • The current backend unit format is the initial contract: token text pool, token table, declaration table, and function table in TLV form.
  • Keep runtime intrinsics hidden behind frontend lowering. User code should not need to know backend helper names.
  • Excluded language features for this pass remain goroutines, channels, select, cgo, and generics. Everything else covered by rtg_tests is in scope.
  • When a backend miscompile is found, stop frontend work, add a minimal tests/ regression, root cause it, and fix it if it blocks the pass.

Architecture

  1. Driver: implement rtg ./cmd/app, -t os/arch, -s, -o output, diagnostics, and deterministic exit codes.
  2. Loader: resolve the nearest go.mod, package directories, import paths, files, and deterministic package order without invoking the host Go toolchain.
  3. Parser: parse the in-scope Go syntax needed by rtg_tests and the frontend source itself.
  4. Checker: resolve packages, names, constants, types, methods, selector rules, assignments, calls, conversions, builtins, and initialization order.
  5. Lowerer: translate checked Go into the backend unit model, adding generated helpers for frontend-only semantics.
  6. Linker: combine package units in memory, resolve generated symbols, order package initialization, and emit one final backend unit.
  7. Backend bridge: call RTGX in memory when possible, with a CLI fallback only for debugging.

Implementation Order

  1. Create the smallest rtg/cmd/rtg that accepts the final CLI and can compile a single-package hello world through the existing binary unit path.
  2. Implement deterministic file and module loading for rtg ./cmd/app, including multiple files in one package.
  3. Implement package graph loading and linking for local module imports before adding broad language features.
  4. Bring up declarations, functions, variables, constants, scalar types, strings, slices, pointers, structs, methods, and package initialization.
  5. Pass rtg_tests/quick by category, keeping every feature vertical: parser, checker, lowering, runtime support, and diagnostics land together.
  6. Reach stage1 self-hosting as soon as the frontend can compile its own source shape, even if the extended corpus is not complete yet.
  7. Reach stage2 and stage3 self-hosting, then keep stage3 in the loop for every following feature group.
  8. Finish the extended corpus categories: arrays, maps, interfaces, function values, closures, dynamic dispatch, defer, panic, recover, unsafe, conversions, and multi-package interactions.
  9. Run the full acceptance set with both host-built and self-hosted frontends.
  10. Optimize only after correctness is stable; performance work must preserve the self-host and full corpus gates.

Feature Scope

  • Required: modules, packages, imports, package initialization, multiple files per package, and deterministic linking.
  • Required: constants, variables, functions, methods, structs, embedded fields, promoted selectors, arrays, slices, strings, pointers, maps, interfaces, function values, closures, and dynamic dispatch.
  • Required: defer, panic, recover, builtins used by the corpus, conversions, and unsafe forms used by the corpus.
  • Required: clear diagnostics for unsupported or malformed input, especially excluded features.
  • Excluded for now: goroutines, channels, select, cgo, and generics.

Testing Plan

  1. Keep go test ./rtg_tests passing at all times. This runs host validation and, once rtg/cmd/rtg exists, frontend validation.
  2. Run targeted category tests while implementing features, for example go test ./rtg_tests -run TestFrontendQuickCorpus/functions.
  3. Run RTG_FRONTEND_EXTENDED_TESTS=1 go test ./rtg_tests before considering any feature group complete.
  4. Run stage tests explicitly: host stage0 to stage1, stage1 to stage2, stage2 to stage3, then stage3 against quick and extended corpus.
  5. Run go test -run '^TestFrontendCompilerPerformance$' . once self-hosting exists.
  6. Run top-level backend checks before merging frontend changes: go test -run '^TestCompilerPerformance$' . and go test -timeout 30m -skip '^TestCompilerPerformance$' ..

Self-Hosting Discipline

  • Do not postpone self-hosting until the end. Start compiling the frontend with itself as soon as a minimal subset is available.
  • Keep frontend source simple enough for the compiler being built: explicit loops, simple data structures, deterministic allocation, and minimal dependencies.
  • When stage1, stage2, or stage3 fails, reduce the failure to a small rtg_tests case or backend tests regression before continuing feature work.
  • Track stage output size, wall time, and max RSS after every major feature group.

Performance Budget

The hard gate for this pass is already encoded in main_test.go: stage2 must build stripped stage3 within 500ms, 32MB max RSS, and 1MB binary size, using the best of three attempts. Treat those as design constraints, not cleanup targets.

  • Prefer compact typed data structures over reflection-like generic representations.
  • Avoid repeated whole-program scans after loading; cache package, symbol, and type indexes.
  • Keep unit emission deterministic and streaming-friendly.
  • Measure before optimizing; compare against current baselines for unit decode, backend parse, and full self-host time.

Non-Goals For This Pass

  • No goroutines, channels, select, cgo, or generics.
  • No speculative standard library expansion.
  • No broad rewrite of the backend unit format unless the current format blocks correctness or self-hosting.
  • No source-emission frontend fallback. The frontend must exercise the binary unit path.
