RTG Frontend Plan

Add a source-to-source compiler under rtg/ that accepts ordinary Go source within an intentionally restricted feature set, parses complete packages into RTG-owned ASTs, lowers each package into a stable platform-independent unit, and then compiles and links those units with the existing retargetable rtgx backend.

Goal

The current compiler is a backend for a small Go-like subset. The new rtg/ frontend should become the user-facing compiler layer: it reads normal Go files, understands multiple source files and packages, performs frontend validation and lowering, and emits backend-compatible RTG units that can be compiled in memory or distributed as stable artifacts.

source to source package units stable unit artifacts in-memory rtgx backend separate linker self-hosting frontend

Non-Goals For The First Frontend

Still Excluded

  • Generics
  • Goroutines and the go statement
  • Channels and select
  • Interfaces, type assertions, and type switches
  • Reflection and dynamic dispatch
  • Full standard library compatibility

Deferred

  • Closures and function values
  • Maps
  • defer, panic, and recover
  • External module fetching beyond simple local module resolution
  • Garbage collection or general heap object tracing

Allowed Direction

  • Reject unsupported Go with precise diagnostics.
  • Desugar supported Go into RTG subset source.
  • Keep emitted units independent of target OS and architecture.
  • Keep the frontend implementation self-hostable and avoid standard library dependencies.
  • Let the backend own machine code and the linker own executable formats.

Proposed Directory Layout

rtg/
  cmd/rtg/             command line frontend
  scan/                self-hostable Go lexer for the supported source surface
  parse/               hand-written parser that builds RTG-owned ASTs
  mod/                 simple first-stage go.mod parsing and import resolution
  load/                package/file grouping and module-aware source loading
  check/               subset validation and name/type checking
  lower/               AST-to-unit normalization and desugaring
  unit/                stable platform-independent unit model and encoder
  link/                package-unit linker and entrypoint/runtime assembly
  std/                 RTG-compatible parallel standard tree
  rtgx/                in-memory backend API and command wrapper
  testdata/            frontend fixture programs and golden lowered output

The directory names are intentionally boring: each package has one job, and the existing backend remains the code generator used by rtg/rtgx.

Pipeline

Stage Responsibility Output
Load Implement simple module-aware loading: parse the module path from go.mod, resolve local module imports, group files by package, and load needed dependencies from rtg/std. Loaded package graph with stable paths and byte content.
Parse Use a hand-written lexer and parser so the frontend can self-host without depending on go/parser, go/ast, or go/types. RTG-owned AST files and source positions.
Check Perform name/type validation for the supported subset and produce diagnostics for excluded features. Typed AST metadata and package export metadata, or a diagnostic list.
Lower Normalize each package into backend-shaped constructs: explicit temporaries, stable symbol names, package export records, and simple control flow where useful. One platform-independent unit per package.
Emit Print deterministic RTG subset source units that are stable enough to cache, distribute, diff, and feed back into the linker. One distributable RTG subset source unit per package.
Compile Hand package units to rtgx through an in-memory API and compile target-independent units into target-specific objects. Target object records with symbols and relocations.
Link Resolve package symbols, add runtime/startup code, lay out data/code, and write the final executable or WASI module. Target executable or WASI module.

Compile Unit Model

  • A package unit is all supported files belonging to one package.
  • Each package is lowered into exactly one RTG unit artifact.
  • All files in a package share one namespace after Go name resolution.
  • Top-level declarations can be spread across files and referenced before declaration.
  • Package imports resolve through the module system to other RTG package units.
  • Standard-library imports resolve to RTG-compatible packages in rtg/std.
  • The emitted unit must not depend on original file order except for deterministic output.
  • The linker combines package units and owns cross-package symbol resolution.

In-Memory Backend

Using rtgx in memory is feasible and should be the preferred path. The current command-line backend can remain as a wrapper, but the core should expose an API that accepts already-loaded RTG subset source units and returns target object records without requiring temporary source files. 

RTG subset source unit + target
  -> rtgx compile
  -> object { code, data, symbols, relocations, target metadata }
  -> linker
  -> executable or module bytes
  • The frontend can still emit RTG subset source unit files with -emit-unit for caching and distribution.
  • The normal compile path should keep units in memory and avoid extra filesystem round trips.
  • The backend API should be deterministic for the same unit source, target, and compiler version.
  • The CLI should use the same API as tests, so in-memory and file-backed builds cannot drift.

Lowering Rules

Names

  • Give every package-level symbol a stable package-qualified unit name.
  • Preserve exported/unexported spelling only when it matters for diagnostics.
  • Generate stable temporary names that cannot collide with user names.

Types

  • Map supported Go scalar, pointer, slice, string, and struct types into RTG subset types.
  • Reject unsupported array, map, interface, channel, and function value types.
  • Preserve named types where the backend already supports them.

Expressions

  • Lower complex expressions into temporaries when backend evaluation order is fragile.
  • Rewrite supported builtins such as len, append, copy, and make into backend-compatible forms.
  • Hide runtime intrinsics behind ordinary-looking frontend constructs and rtg/std packages.
  • Reject unsupported builtins with source positions.

Statements

  • Keep ordinary if, for, switch, break, continue, and goto when they fit the backend.
  • Lower range only after explicit support is planned; reject it in phase one.
  • Reject defer, go, and select.

Command Line Shape

rtg -t linux/amd64 -o out ./cmd/app
rtg -emit-unit -o pkg.rtg.go ./pkg/a.go ./pkg/b.go
rtg -link -t wasi/wasm32 -o app.wasm main.rtg.go dep.rtg.go
rtg -check ./input.go
  • -t selects the backend target and defaults to the host-compatible target.
  • -emit-unit stops after emitting stable RTG subset source unit artifacts.
  • -link links previously emitted unit artifacts for a target.
  • -check validates frontend support without invoking the backend.
  • A package directory argument such as rtg ./cmd/app should behave like the normal build entrypoint.
  • The frontend should print diagnostics against original file paths and line numbers.

Host Go Safety

  • Generated unit files use the .rtg.go suffix by convention.
  • The frontend should emit units into a build/cache directory by default, not beside user source files.
  • When unit files are written as files, they should include a restrictive build constraint such as //go:build rtg so the host Go toolchain ignores them by default.
  • rtgx should ignore that build constraint comment and treat the file as RTG subset source.

Milestones

Status Milestone Acceptance Criteria
first Skeleton CLI and loader Can build rtg ./cmd/app, parse the local module path from go.mod, group files into packages, resolve simple local imports and needed rtg/std packages, and emit structured diagnostics for unsupported layouts.
first Self-hostable parser Can lex and parse the supported Go syntax into RTG-owned ASTs without using Go standard library parser packages.
first Subset checker Rejects generics, goroutines, channels, interfaces, maps, function values, and imports with useful source positions.
first Stable package unit emitter Emits one deterministic distributable RTG subset source unit per package for simple programs with globals, functions, structs, and calls across files.
next Expression normalization Introduces temporaries to preserve Go evaluation order before handing code to the backend.
next In-memory backend integration Can compile package unit source through rtgx without temporary files for every supported target.
next Linker stage Can link multiple package objects, resolve symbols, include runtime/startup code, and produce final target binaries.
next Self-hosting frontend The frontend source compiles through rtgx and uses rtgx as its backend.
next Parallel standard tree Provides only the initial rtg/std packages needed by the frontend and early fixtures, expanding later as required.
next Frontend fixture suite Golden lowered output plus end-to-end tests that compare host Go behavior against emitted backend binaries.

Testing Strategy

  • Unit tests for loader grouping, diagnostics, and declaration ordering.
  • Golden tests for package unit artifacts so frontend changes are reviewable and distributable units remain stable.
  • End-to-end tests that compile package units through in-memory rtgx, link them, and compare output with host Go for supported programs.
  • Negative tests for every explicitly excluded feature.
  • Cross-target smoke tests should reuse the existing backend target matrix once the frontend can call rtgx in memory.
  • Self-hosting tests should build the frontend with rtgx and use the result to compile a small multi-package program.

Settled Direction

  • The frontend should grow into full Go module support; stage one keeps module handling simple and starts with the local module path plus local imports.
  • The project will provide its own parallel standard tree in rtg/std, initially limited to what the frontend and early programs need.
  • Each package lowers into one stable platform-independent unit.
  • The stable unit artifact format is RTG subset source with the .rtg.go suffix.
  • Package dependencies remain separate until a linker stage combines them.
  • Stage one should keep existing Go compatibility intentionally simple while preserving the module/package shape.
  • The normal package build form should be rtg ./cmd/app.
  • The frontend should be written in the RTG-supported subset and self-host with rtgx as the backend.
  • The frontend should use hand-written parsing and checking code, avoiding Go standard library compiler packages.
  • Runtime intrinsics should be hidden from user code and may be expanded as long as each intrinsic is implemented on every supported backend target.
  • Emitted RTG subset source units should be stable enough to cache, commit in golden tests, and distribute.
